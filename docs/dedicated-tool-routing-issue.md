# Agent 专用工具路由问题记录

## 问题

Mycode 已经注册了 `read_file`、`find_files`、`search_code`、`write_file` 和 `edit_file` 等受控工具，但模型仍可能自行构造 Shell 命令并调用 `run_command`。

已观察到的例子：用户要求查看目录内容时，模型选择执行 `ls -la`，没有优先使用项目准备好的只读工具。由于 `run_command` 被统一视为有副作用工具，该只读场景又触发了不必要的权限确认。

## 原因

- 当前框架把全部已注册工具描述发送给模型，由模型自行选择工具。
- “优先使用专用工具”目前只是提示词约束，不是执行层的强制规则。
- `find_files` 等专用工具与 `ls -la` 的输出能力并不完全相同，模型可能判断通用命令更直接。
- `run_command` 不区分只读命令和有副作用命令，任何 Shell 调用都进入相同的权限流程。

## 判断

“Agent 应使用项目预先准备好的工具，而不是默认让模型临时编写命令”这一方向是正确的。专用工具具有稳定参数、明确工作区边界、结构化结果和可测试行为，更容易实施权限控制，也能减少 Shell 拼接、命令注入、平台差异和模型误判风险。

但不建议完全删除 `run_command`。运行测试、构建、格式化、版本控制及项目脚本等操作无法全部由现有文件工具覆盖。`run_command` 应定位为受控兜底能力，而不是与专用工具地位相同的默认入口。

## 建议原则

1. 能由专用工具完成的读取、搜索、查找、写入和编辑操作，必须优先使用专用工具。
2. 专用工具应由框架直接识别为只读或有副作用，而不是让模型自行声明安全属性。
3. `run_command` 只在专用工具无法完成任务时使用，并继续经过黑名单、路径沙箱、规则和人工审批。
4. 如果专用工具能力不足，应优先补齐专用工具，例如提供结构化目录列表工具，而不是长期依赖 `ls`。
5. 不能只依赖提示词保证工具选择；执行层应能拒绝或改写可由专用工具替代的 Shell 调用。
6. 不应把“模型声称命令只读”作为自动放行依据。

## 建议验收行为

- 查看已知文件时调用 `read_file`，不调用 `cat`、`sed` 等 Shell 命令。
- 查找文件时调用 `find_files`，不调用 `find`、`ls` 等 Shell 命令。
- 搜索代码时调用 `search_code`，不调用 `grep`、`rg` 等 Shell 命令。
- 修改文件时调用 `write_file` 或 `edit_file`，不使用重定向、`sed -i` 等 Shell 写法。
- 运行测试、构建或项目脚本时允许使用 `run_command`，并保留权限防护。
- 当专用工具无法表达用户需要的目录元数据时，系统应明确使用受控兜底，或新增结构化目录工具。

## 待决策

- 是否在执行层硬性禁止可由专用工具替代的 `run_command`。
- 是否新增 `list_directory`，覆盖隐藏文件、权限、大小和类型等目录信息。
- 专用只读工具是否完全绕过权限规则，还是仅绕过人工确认但继续执行工作区沙箱检查。
