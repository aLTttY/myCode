# 终端中文输入删除问题技术复盘

## 背景

myCode 当前交互式 CLI 使用 Python 内置 `input("myCode> ")` 读取用户输入。用户在终端中输入中文问题后再使用 Backspace 删除时，出现屏幕显示残留或删除不符合预期的问题。

复现场景：

```text
输入：我叫什么名字
操作：连续按 Backspace 6 次
现象：屏幕上仍残留类似“我叫什”的内容
```

## 影响范围

- 影响终端输入阶段的编辑体验。
- 不影响 Provider 请求、模型响应、流式输出解析或会话历史管理。
- 不影响非交互式测试中通过 mock `builtins.input` 注入的输入。

## 当前实现

相关代码位于 `src/mycode/cli.py`：

```python
user_text = input("myCode> ").strip()
```

当前实现没有引入专门的终端行编辑库，因此中文输入、宽字符显示、Backspace 行编辑等行为主要依赖底层终端和 Python 默认输入机制。

## 根因分析

中文字符在终端里涉及两个不同维度：

- 编码层：中文字符在 UTF-8 中通常占多个字节。
- 显示层：中文字符在终端中通常占两个显示列宽。

裸 `input()` 对真实 TTY 的行编辑能力有限。当用户按 Backspace 时，终端需要同时维护输入缓冲区、光标位置和屏幕回显。如果底层行编辑器不能正确处理 Unicode 宽字符，就会出现以下问题：

- 实际输入缓冲和屏幕显示不同步。
- 删除操作按字节、列宽或字符边界处理不一致。
- 屏幕上看起来残留了已经删除或未正确擦除的中文字符。

因此该问题本质上不是业务逻辑 bug，而是 CLI 交互层缺少可靠的 Unicode 行编辑能力。

## 方案比较

| 方案 | 说明 | 优点 | 缺点 |
|------|------|------|------|
| 导入 `readline` | 在 CLI 启动时启用 readline/libedit 行编辑 | 改动小，不新增第三方依赖 | macOS 上常见的是 libedit，中文宽字符行为仍可能不一致 |
| 使用 `prompt_toolkit` | 用成熟交互式命令行库替代裸 `input()` | Unicode 和宽字符处理更可靠，支持历史记录、快捷键和更好的提示符能力 | 新增运行时依赖，CLI 代码需要轻微调整 |
| 自行处理 TTY | 手动进入 raw mode，解析按键、维护光标和缓冲 | 依赖最少，可完全控制行为 | 复杂度高，容易引入终端兼容性问题，不符合当前 MVP 的工程收益 |

## 方案选择

选择 `prompt_toolkit`。

理由：

- 该问题属于终端交互基础能力，不适合在业务代码中自行处理字节、字符宽度和光标控制。
- `prompt_toolkit` 是成熟的 Python CLI 输入库，能更系统地处理 Unicode、中文宽字符、Backspace、方向键和历史输入。
- 相比只导入 `readline`，`prompt_toolkit` 的跨平台一致性更好。
- 相比自行处理 TTY，维护成本和出错风险显著更低。

## 建议实施范围

第一阶段只解决输入稳定性，不扩展复杂 TUI。

建议变更：

1. 在 `pyproject.toml` 增加运行时依赖 `prompt_toolkit`。
2. 在 CLI 层封装一个输入函数，例如 `read_user_input(prompt: str) -> str`。
3. 用 `prompt_toolkit.prompt()` 替代裸 `input()`。
4. 保留现有退出命令、空输入跳过和异常处理语义。
5. 增加伪终端或函数级测试，覆盖中文输入删除的关键路径。

## 验收标准

- 用户输入 `我叫什么名字` 后，连续 Backspace 删除时，屏幕显示和实际输入缓冲保持一致。
- 删除中文字符时不会出现残留字符、错位光标或异常蜂鸣。
- 现有 CLI 退出命令仍可用：`exit`、`quit`、`退出`。
- 现有单元测试继续通过。
- 新增测试能覆盖 CLI 输入函数使用 `prompt_toolkit` 的行为。

## 风险和注意事项

- `prompt_toolkit` 会新增依赖，需要更新安装说明。
- 单元测试不应依赖真实用户交互，应通过封装输入函数或 monkeypatch `prompt_toolkit.prompt` 测试。
- 如果后续引入多行输入、自动补全或命令模式，需要单独设计，不应混入本次修复。
