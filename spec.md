# myCode Agent Loop Spec

## 背景

myCode 当前已经具备命令行多轮对话、六个核心本地工具、工具注册中心、流式工具调用解析、工具执行和一次工具结果回灌能力。模型已经可以读取文件、写文件、改文件、执行命令、查找文件和搜索代码，但当前闭环仍然是“一步一停”：一次用户输入最多触发一轮工具调用，工具结果回灌后本轮停止。

这导致 myCode 还不能自主完成较长任务。模型看到工具结果后，即使知道下一步该继续读取、修改、测试或修正，也必须等待用户再次催促。用户体验仍接近“带工具的聊天”，不是“能自主推进任务的 Agent”。

本阶段目标是在已有工具系统之上增加 Agent Loop，让 myCode 能按 ReAct 风格持续循环：调用模型、执行工具、观察结果、调整下一步，直到模型不再请求工具或触发明确停止条件。同时提供异步事件流，让 Agent 执行过程和终端界面解耦，为后续更好的 UI、权限系统和上下文压缩留下空间。

## 目标

- 支持 ReAct 风格自主循环，让 myCode 能连续多轮调用模型和工具，直到任务完成或停止。
- 支持明确停止条件，避免无限循环和不可控执行。
- 提供统一异步事件流，输出文本、工具调用、工具结果、Token 用量和进度事件。
- 支持流式双路收集：一边实时把文本增量推给界面，一边保留完整响应供循环判断。
- 支持一次模型响应中的多个工具调用，并按安全性分批执行。
- 支持 Plan Mode 两段式流程：`/plan` 只允许读类工具生成计划，`/do` 再使用完整工具集合执行。
- 保持范围克制：本阶段不实现权限系统、上下文压缩或用户交互式确认。

## 功能需求

- F1: myCode 应支持 Agent Loop：一次用户请求可以触发多轮“模型响应 → 工具执行 → 结果回写 → 再次模型响应”的循环。
- F2: Agent Loop 应在模型不再请求工具并输出最终回复时正常停止。
- F3: Agent Loop 应支持最大迭代次数；达到上限时停止，并向用户说明停止原因。
- F4: Agent Loop 应支持用户取消；取消后停止继续调用模型和工具，并向用户说明已取消。
- F5: Agent Loop 遇到连续未知工具调用时应停止，避免模型反复请求不存在的工具。
- F6: Agent Loop 遇到模型流式响应错误或工具调用流解析错误时应停止，并返回可理解错误。
- F7: 每轮模型响应中的文本增量应实时推送给界面，不等完整响应结束。
- F8: 每轮模型响应也应被完整收集，用于判断是否有工具调用、是否完成、是否需要继续循环。
- F9: myCode 应提供统一异步事件流，至少包含文本增量、工具调用开始、工具结果、Token 用量和循环进度事件。
- F10: Agent 执行过程不应直接绑定终端输出；CLI 或其他界面只消费事件流并决定如何展示。
- F11: 当模型一次返回多个工具调用时，myCode 应能识别全部工具调用。
- F12: 多个工具调用应按安全性分批执行：读类、无副作用工具可以并发执行；写文件、改文件、执行命令等有副作用工具应串行执行。
- F13: 工具执行结果应按原工具调用关联关系回写到对话历史，确保模型能正确理解每个工具结果。
- F14: 每轮循环都应向用户或界面报告当前进度，例如当前迭代次数、最大迭代次数、正在执行的工具数量或停止原因。
- F15: Agent Loop 应复用现有工具系统、工具结果格式和工作区安全边界，不重新定义一套工具机制。
- F16: `/plan` 模式应只开放读类工具，让模型可以查看项目并输出执行计划，但不能写文件、改文件或执行命令。
- F17: `/plan` 模式结束时应输出可读计划，并保留足够上下文供后续 `/do` 使用。
- F18: `/do` 模式应切换到完整工具集合，按用户任务或已有计划执行实际修改、命令和验证。
- F19: 普通用户输入在非 `/plan` 模式下应默认使用完整 Agent Loop，而不是上一阶段的一轮工具回灌。
- F20: 当模型没有请求工具时，普通聊天应仍保持现有流式文本输出能力。

## 非功能需求

- N1: Agent Loop 应有清晰边界，不能因模型持续请求工具而无限运行。
- N2: 事件流应稳定、可测试，并能支持未来 UI 折叠工具日志、独立进度面板或更复杂终端体验。
- N3: 读类工具并发执行不应破坏结果关联顺序；有副作用工具串行执行不应产生竞态。
- N4: 停止原因应结构化且可读，便于 CLI 展示和测试断言。
- N5: Plan Mode 应严格限制工具集合，避免 `/plan` 阶段产生文件修改或命令执行副作用。
- N6: Agent Loop 应复用现有 Provider 抽象和工具注册中心，避免把供应商协议或工具细节耦合进 CLI。
- N7: 自动化测试应覆盖正常完成、达到迭代上限、用户取消、未知工具、流错误、多工具批处理和 Plan Mode。
- N8: 文档应明确本阶段仍不包含权限确认、上下文压缩和交互式审批。

## 不做的事

- 本阶段不实现权限系统。
- 本阶段不实现用户交互式确认每一次工具调用。
- 本阶段不实现上下文压缩、历史裁剪或长期记忆。
- 本阶段不实现复杂 TUI、隐藏工具面板、折叠日志 UI 或可视化任务看板。
- 本阶段不实现人工审批计划后自动恢复执行的完整工作流；`/plan` 和 `/do` 只提供两段式命令能力。
- 本阶段不新增 Git 专用工具、网络搜索工具、浏览器工具或外部服务工具。
- 本阶段不改变现有六个核心工具的基本职责。
- 本阶段不保证所有模型供应商的 Token 用量字段都完整可用；不可用时应允许为空或缺省。
- 本阶段不实现跨进程持久化 Agent 运行状态。

## 验收标准

- AC1: 一次用户请求可以触发多轮模型调用和工具执行，直到模型给出不再请求工具的最终回复。
- AC2: 模型不再请求工具时，Agent Loop 正常停止，并输出最终文本。
- AC3: 达到最大迭代次数时，Agent Loop 停止，并产生明确的“达到迭代上限”停止原因。
- AC4: 用户取消时，Agent Loop 停止后不再继续调用模型或工具，并产生明确取消事件。
- AC5: 连续未知工具调用达到阈值时，Agent Loop 停止，并说明未知工具名称和停止原因。
- AC6: 模型流式响应错误或工具调用解析错误时，Agent Loop 停止并输出可理解错误。
- AC7: 模型流式文本增量能实时作为事件发出，同时完整响应可用于后续循环判断。
- AC8: 事件流至少能观察到文本增量、工具调用开始、工具结果、Token 用量和循环进度。
- AC9: CLI 或测试消费者可以只通过事件流观察 Agent 执行过程，不需要直接读取 Agent 内部状态。
- AC10: 一次返回多个读类工具调用时，myCode 能并发执行并正确回写每个结果。
- AC11: 一次返回多个有副作用工具调用时，myCode 按顺序串行执行并正确回写每个结果。
- AC12: 混合工具调用会按安全性分批执行，读类工具与有副作用工具不会无序交叉造成竞态。
- AC13: 工具结果按对应工具调用 ID 回写到对话历史，模型能在下一轮看到所有结果。
- AC14: 每轮循环都会产生进度事件，包含当前迭代次数和最大迭代次数。
- AC15: `/plan` 模式下模型只能使用读类工具；写文件、改文件和执行命令不会被开放给模型。
- AC16: `/plan` 模式能基于读类工具结果输出可读计划。
- AC17: `/do` 模式能使用完整工具集合执行实际任务。
- AC18: 普通输入默认进入完整 Agent Loop，而不是停留在单轮工具回灌。
- AC19: 没有工具调用的普通聊天仍保持流式输出。
- AC20: 自动化测试覆盖正常完成、迭代上限、取消、未知工具、流错误、多工具批处理、Plan Mode 和普通聊天回归。
