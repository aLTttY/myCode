# Structured System Prompt Spec

## 背景

myCode 当前已经具备命令行多轮对话、基础工具系统、Agent Loop、Plan Mode 和多 Provider 流式调用能力。现有运行时主要把用户输入、工具结果和助手回复作为普通对话历史传给模型，模式约束也以用户消息前缀形式拼入。全局行为约束仍然过于简略，稳定指令、工具描述、动态环境信息和会话级临时规则没有清晰分层。

这会带来三个问题：第一，模型缺少足够明确的身份、行为、工具、安全和输出规则，实际执行质量不稳定；第二，稳定的大段指令和动态信息混在一起，不利于利用模型 API 的提示缓存能力；第三，Plan Mode 等会话级规则按普通用户内容注入，既污染缓存，也容易被模型当成用户请求的一部分回复。

本阶段目标是把原来的全局指令扩展为结构化系统提示：稳定规则走可缓存通道，环境信息和按轮变化的动态规则走系统级补充消息通道，并通过 API 返回的缓存命中字段验证缓存策略生效。该阶段只覆盖系统提示结构、注入策略、工具描述强化和人工对比验证准备，不引入项目指令文件加载、自动记忆、真实 MCP 接入或自动化评估体系。

## 目标

- 建立结构化系统提示，把全局指令按职责拆分并按优先级稳定拼装。
- 明确区分稳定内容和动态内容，让稳定指令与工具描述可被模型 API 缓存，环境信息和会话级补充指令不污染稳定缓存。
- 用系统级补充消息承载运行中动态指令，避免把模式规则伪装成用户输入。
- 在全局指令和工具描述中双重强化关键工具使用约定，提高模型对专用工具、读取优先、编辑前观察等规则的遵守率。
- 支持会话级开关按轮次注入，兼顾规则可见性和 token 成本。
- 记录并展示 API 缓存命中信息，用典型场景支持人工定性对比。

## 功能需求

- F1: myCode 应提供结构化全局指令，由固定模块按优先级拼装，模块至少覆盖身份、系统约束、任务模式、动作执行、工具使用、语气风格和文本输出。
- F2: 固定模块应保持稳定顺序，整体按“身份、系统约束、任务模式、动作执行、工具使用、语气风格、文本输出、环境信息”的优先级从前往后组织。
- F3: myCode 应支持可选模块追加在固定模块之后，可选模块至少预留自定义指令、已激活的 Skill 和长期记忆的位置。
- F4: 模块之间应使用清晰分隔，便于阅读、测试和后续插入新模块。
- F5: myCode 应区分稳定提示内容和动态提示内容；稳定全局指令和工具描述应进入可缓存通道，环境信息、对话历史和按轮变化的补充指令应进入消息通道。
- F6: 环境信息应作为系统级内容注入，至少能表达当前工作目录、日期时间或运行模式等运行时上下文，但不应改变稳定提示缓存键。
- F7: myCode 应支持一种带特殊标签的系统级补充消息，用于注入运行中的动态指令，并明确该消息不是用户输入。
- F8: 会话级模式规则应通过系统级补充消息注入，而不是拼接到用户文本中。
- F9: 会话级开关的注入应支持轮次策略：首次出现时完整注入，间隔若干轮重复完整规则，其余轮次注入精简提醒。
- F10: Plan Mode 的动态指令应明确只允许观察、分析和产出计划，不允许写文件、改文件或执行命令。
- F11: Do Mode 或默认执行模式的动态指令应明确允许在规则约束下使用完整工具集推进任务。
- F12: 工具描述应强化“优先使用专用工具”的规则，例如读文件使用读文件工具、找文件使用查找工具、搜索代码使用搜索工具、修改文件使用编辑工具。
- F13: 工具描述和全局指令应共同强化“编辑前必须先读取或搜索确认当前内容”的规则。
- F14: 工具描述和全局指令应共同强化工作区安全边界，模型不应尝试越界读取、写入或执行无关命令。
- F15: Provider 层应能把稳定系统提示、动态系统补充消息、普通对话历史和工具描述转换为各模型 API 可接受的请求形态。
- F16: 对支持缓存控制的模型 API，myCode 应在稳定提示和工具描述上使用可缓存标记或等价机制。
- F17: 对不支持缓存控制或不返回缓存字段的模型 API，myCode 应保持正常对话能力，并以可理解方式标明无法验证缓存命中。
- F18: myCode 应解析模型 API 返回的缓存命中相关字段，并把缓存读取、缓存创建或等价指标纳入 token 用量事件或可观察输出。
- F19: myCode 应准备一组典型人工对比场景，至少覆盖读取后编辑、Plan Mode 只读、优先专用工具、环境信息变化但稳定缓存不变、连续多轮动态规则注入。
- F20: 现有普通聊天、工具调用、Agent Loop 和 Plan/Do 命令入口在引入结构化系统提示后应保持可用。

## 非功能需求

- N1: 系统提示模块应可测试、可扩展，新增模块时不需要重写 Provider 或 Agent Loop 的核心流程。
- N2: 稳定提示内容应尽量保持确定性，避免把当前时间、工作目录、用户输入、对话历史等动态内容混入缓存段。
- N3: 动态补充指令应在模型语义上具备系统级优先级，不能被当作用户请求回复。
- N4: 注入频率应可控，避免每轮重复完整模式说明造成不必要 token 成本。
- N5: 缓存命中解析应兼容不同 Provider 返回字段差异，未知字段不能导致主流程失败。
- N6: 关键规则应在提示和工具描述中表达一致，避免同一约定在不同位置互相冲突。
- N7: 自动化测试应覆盖提示拼装、稳定/动态分离、动态补充消息、模式注入频率、工具描述强化、缓存字段解析和现有回归路径。

## 不做的事

- 本阶段不实现项目级指令文件加载。
- 本阶段不实现自动记忆写入、召回或长期记忆管理。
- 本阶段不接入真实 MCP 工具或 MCP 服务发现。
- 本阶段不实现自动化评估、打分器或批量 benchmark。
- 本阶段不新增复杂权限审批系统。
- 本阶段不改变现有六个核心工具的基本执行能力。
- 本阶段不要求所有 Provider 都真实支持提示缓存；不支持时只需优雅降级和可观察说明。

## 验收标准

- AC1: 程序能生成包含七个固定模块的结构化全局指令，模块顺序符合身份、系统约束、任务模式、动作执行、工具使用、语气风格、文本输出。
- AC2: 环境信息被放在固定模块之后或动态系统内容中，且改变环境信息不会改变稳定提示内容。
- AC3: 自定义指令、已激活 Skill、长期记忆作为可选模块有明确追加位置，不影响固定模块顺序。
- AC4: 生成的提示文本模块之间有稳定分隔，快照或单元测试能检测顺序和内容变化。
- AC5: Provider 请求中稳定全局指令和工具描述与环境信息、对话历史、模式补充消息分离。
- AC6: 运行时动态补充指令使用特殊标签，并以系统级内容发送，不出现在用户消息正文中。
- AC7: Plan Mode 请求不再把模式说明拼接进用户文本，而是通过系统级补充消息表达。
- AC8: Plan Mode 首轮包含完整规则，间隔轮次重复完整规则，其他轮次只包含精简提醒。
- AC9: Plan Mode 下写文件、改文件和执行命令仍不会开放给模型，且动态指令明确说明该限制。
- AC10: Do Mode 和默认模式仍能开放完整工具集，并通过动态指令表达执行模式约束。
- AC11: 工具描述中能观察到专用工具优先规则。
- AC12: 工具描述和全局指令中都能观察到编辑前读取或搜索确认的规则。
- AC13: 工具描述和全局指令中都能观察到工作区边界和安全规则。
- AC14: 支持缓存控制的 Provider 请求包含稳定提示或工具描述的缓存标记或等价结构。
- AC15: 不支持缓存控制的 Provider 能正常完成请求，并产生“缓存不可验证”或等价可观察信息。
- AC16: API 返回缓存命中字段时，myCode 能解析并通过 token 用量事件或终端输出展示缓存读取、创建或等价指标。
- AC17: 人工对比场景文档或测试夹具覆盖读取后编辑、Plan Mode 只读、专用工具优先、环境变化缓存稳定、多轮动态注入。
- AC18: 普通无工具聊天仍保持流式输出。
- AC19: 现有 Agent Loop 工具调用和工具结果回写仍可正常运行。
- AC20: 自动化测试覆盖结构化提示、动态注入、缓存字段解析和现有 Plan/Do 回归。
