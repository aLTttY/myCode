# Mycode 五层权限系统 Spec

## 背景

Mycode 已具备工具系统和可持续迭代的 Agent Loop，但工具调用目前缺少统一的执行前权限判定。文件工具虽有基础工作区路径检查，命令工具仍可直接交给 shell 执行；系统也没有分层规则、权限模式或用户审批机制。权限拒绝若不能作为可观察结果返回模型，还会削弱 Agent 自行调整策略的能力。

## 目标

- 在所有工具执行前建立统一的五层防御：不可绕过的危险命令黑名单、路径沙箱、可配置规则、权限模式和人在回路。
- 支持用户级、项目级、本地级与会话级规则，并按越靠近当前会话优先级越高的顺序判定。
- 支持严格、默认、放行三档权限模式，为规则未命中的调用提供一致行为。
- 支持本次、本会话和永久三种交互放行范围。
- 权限拒绝以结构化工具结果回灌 Agent Loop，使模型可以调整方案，而不是直接终止任务。

## 功能需求

- F1: 所有工具调用在实际执行前都必须经过统一权限判定；未通过判定的调用不得触达工具实现。
- F2: 系统必须在其他权限机制之前检查不可配置放开的危险命令黑名单，拦截灾难性文件系统破坏、磁盘覆写或格式化、系统目录递归权限修改、关机重启和 fork bomb。
- F3: 黑名单不得被权限模式、配置规则或人工审批覆盖；高风险但可能合理的开发操作不进入硬黑名单，交由后续权限层判定。
- F4: 专用文件工具产生的文件读取和写入必须限制在当前项目目录内；路径判定前必须解析规范路径和符号链接，防止通过绝对路径、`..` 或符号链接逃逸。
- F5: 命令工具必须检查命令文本中可识别的显式路径；显式路径必须经过规范化和符号链接解析，确认位于当前项目目录内后才可执行。命令运行时未在参数中显式表达的隐式文件访问不属于本阶段的强制隔离范围。
- F6: 系统必须支持 `工具名(模式)` 形式的 allow/deny 规则，工具名使用真实注册名称，模式支持精确匹配和 glob 匹配。
- F7: 每种工具必须具有稳定的权限目标：命令工具匹配完整命令，文件工具匹配规范化后的工作区相对路径，查找工具匹配路径或 glob 参数，代码搜索工具匹配搜索范围路径；内容和替换文本不参与匹配。
- F8: 规则分别来自用户级、项目级、本地级 YAML 和会话内存；优先级为“会话 > 本地 > 项目 > 用户”。
- F9: 跨层判定时，只采用最高优先级中存在匹配的规则层；同层多规则命中时，精确匹配优先于 glob，同等匹配类型下 deny 优先于 allow。
- F10: 系统必须支持 strict、default、allow 三档权限模式，作为所有配置规则均未命中时的默认行为。
- F11: strict 模式拒绝未命中调用；default 模式把未命中调用交给用户确认；allow 模式放行未命中调用。显式 deny 始终生效，黑名单与路径沙箱在所有模式下均不可绕过。
- F12: default 模式需要审批时，用户可以拒绝，或选择本次、本会话、永久放行。
- F13: 本次放行只适用于当前工具调用；本会话放行生成仅在当前进程有效的会话规则；永久放行写入项目本地私有 YAML。
- F14: 人工审批通过可注入接口提供；交互式 CLI 必须展示工具名、权限目标和判定原因，并收集用户决定。
- F15: 非交互环境没有可用审批接口时，待审批调用必须安全拒绝，不得默认放行。
- F16: 黑名单、沙箱或显式 deny 导致的拒绝不得再次请求用户覆盖。
- F17: 权限拒绝必须作为结构化工具结果写回对话历史，并继续 Agent Loop，让模型能够改用安全命令、专用工具或项目内路径。
- F18: 用户拒绝审批同样不得终止整个 Agent Loop；只有既有迭代上限、取消或其他既有停止条件可以结束任务。
- F19: 权限配置解析失败时必须给出可理解错误，且不得在配置含义不确定时降级为放行。

## 非功能需求

- N1: 权限判定必须集中、确定且可测试，同一工具调用和同一规则集合应产生相同结果。
- N2: 默认行为必须遵循 fail closed；配置无效、显式路径解析失败、显式路径越界或审批不可用时不得自动执行。
- N3: 权限层应独立于具体 Provider，不因模型供应商或流式协议不同而改变安全语义。
- N4: 权限拒绝结果不得泄露配置文件中的敏感内容，也不得回显无需展示的大段文件内容或命令数据。
- N5: 用户审批提示必须清楚区分不可覆盖拒绝、规则拒绝和待用户决定，避免让用户误以为硬安全层可以放开。
- N6: 会话规则只在当前进程生命周期内有效；永久规则写入时必须保持 YAML 有效，且不能破坏已有本地规则。
- N7: 多个有副作用的工具调用必须维持既有串行执行语义，避免并发审批或规则写入造成竞态。
- N8: 引入权限系统后，普通聊天、Plan Mode、Do Mode、工具结果回灌、取消和迭代上限等现有行为应保持可用。
- N9: 自动化测试必须覆盖黑名单、符号链接逃逸、规则优先级、匹配冲突、三档模式、三种放行范围、无交互安全拒绝以及拒绝后继续 Agent Loop。
- N10: 权限配置和错误信息应具备可读性，便于用户理解某次调用为何放行、拒绝或请求确认。

## 不做的事

- 本阶段不限制网络请求或按域名、协议、端口配置网络权限。
- 本阶段不实现 CPU、内存、进程数、执行时长之外的新资源配额。
- 本阶段不实现权限决策审计日志、历史查询或审计导出。
- 本阶段不提供图形化权限管理界面。
- 本阶段不自动把永久放行写入项目共享规则或用户全局规则。
- 本阶段不允许通过配置扩展、删除或覆盖灾难性命令黑名单。
- 本阶段不把普通高风险开发操作全部列入硬黑名单；它们由规则、模式和人工审批控制。
- 本阶段不新增网络沙箱或容器化运行环境。
- 本阶段不通过操作系统沙箱或容器拦截命令运行时的隐式文件访问；这一限制必须作为后续强化项保留在技术设计中。

## 验收标准

- AC1: 任一已注册工具在执行前都会产生明确的权限判定；被拒绝的调用不会进入工具实现。
- AC2: 代表性的根目录破坏、磁盘覆写或格式化、系统目录递归权限修改、关机重启及 fork bomb 命令均被硬拒绝。
- AC3: strict、default、allow 模式、所有层级的 allow 规则以及人工审批均不能放行黑名单命令。
- AC4: 高风险但可能合理的普通开发命令不因硬黑名单直接拒绝，而是继续进入规则、模式或审批判定。
- AC5: 文件工具使用绝对路径、`..` 越界路径或指向项目外部的符号链接时均被拒绝，且不会读写目标。
- AC6: 命令工具包含可识别的项目外显式路径、无法规范化的显式路径或指向项目外部的符号链接时不会执行；拒绝结果明确建议采用项目内路径或专用工具。
- AC7: 精确规则只匹配完全一致的权限目标，glob 规则能匹配预期目标且不误匹配无关目标。
- AC8: `run_command(git *)` 能按完整命令匹配；文件规则按规范化项目相对路径匹配；文件内容和搜索文本的变化不会改变路径规则结果。
- AC9: 会话、本地、项目、用户四层同时存在冲突规则时，实际结果符合“会话 > 本地 > 项目 > 用户”。
- AC10: 同一层存在精确规则和 glob 规则时采用精确规则；同等匹配类型同时存在 allow 与 deny 时采用 deny。
- AC11: 所有配置规则均未命中时，strict 模式拒绝、default 模式请求审批、allow 模式放行。
- AC12: 任意显式 deny 命中后都不会进入人工审批，也不会因 allow 模式而放行。
- AC13: default 模式的审批提示可观察到工具名、权限目标、请求原因，以及拒绝、本次、本会话、永久四种决定。
- AC14: 选择“本次放行”后只有当前调用获准，后续相同调用仍按正常流程重新判定。
- AC15: 选择“本会话放行”后，相同权限目标在当前进程内无需重复审批；重启进程后该规则消失。
- AC16: 选择“永久放行”后，本地私有 YAML 出现有效规则；重启后相同调用仍能命中，项目共享和用户全局配置不被修改。
- AC17: CLI 可以完成真实交互审批；未提供审批接口的底层或非交互调用会安全拒绝待审批操作。
- AC18: 黑名单拒绝、路径沙箱拒绝、显式规则拒绝和用户拒绝都以结构化工具结果回写对话历史。
- AC19: 权限拒绝后 Agent Loop 仍可进入下一次模型迭代，模型能够选择更安全的命令、专用工具或项目内路径继续任务。
- AC20: 多个有副作用工具调用继续串行判定和执行，不出现并发审批提示或永久规则写入冲突。
- AC21: 用户级、项目级和本地级 YAML 均可被加载；不存在的可选配置层不会导致启动失败。
- AC22: YAML 语法错误、未知权限结果、无效规则格式或未知工具名会产生可理解的配置错误，相关调用不会降级为放行。
- AC23: 普通聊天、Plan Mode 只读工具、Do Mode 完整工具集、用户取消和最大迭代次数等既有路径仍通过回归测试。
- AC24: 自动化测试覆盖不可绕过层、规则与模式判定、三种放行范围、无交互拒绝，以及拒绝后继续 Agent Loop 的端到端场景。
