# myCode 工具系统 Spec

## 背景

myCode 当前已经具备命令行多轮对话、配置化模型后端、流式文本回复和进程内会话历史能力，但模型只能生成文本，不能读取项目、修改文件、执行命令或搜索代码。

本阶段要把 myCode 从普通聊天机器人推进为具备基础行动能力的 Coding Agent：用户提出任务后，模型可以在一次对话响应中请求调用本地工具；程序负责识别工具调用、执行对应工具、把结构化结果回灌到对话历史，让模型基于实际结果给出后续回复或停止。

本阶段只实现“一次工具调用结果回灌”的基础闭环，不实现多轮自动 Agent Loop。也就是说，模型拿到一次工具执行结果后，本轮流程停止；连续自主规划、反复调用多个工具、直到任务完成的能力留到后续阶段。

## 目标

- 提供统一的工具抽象，让文件、命令和搜索能力以一致方式暴露给模型。
- 提供六个核心本地工具：读文件、写文件、改文件、执行命令、按模式找文件、搜索代码内容。
- 提供工具注册中心，用于集中登记、按名称查找，并生成模型 API 可识别的工具声明。
- 支持模型在流式响应中发起工具调用，并正确拼接 JSON 参数碎片。
- 支持执行工具后把结构化结果写回对话历史，让模型看到真实执行结果。
- 支持工具执行超时、失败捕获和清晰错误返回，避免程序崩溃。
- 保持范围克制：本阶段只做单次工具调用闭环，不做自动循环。

## 功能需求

- F1: myCode 应提供统一的工具能力抽象，每个工具都能声明名称、描述、参数约束和可执行行为。
- F2: myCode 应提供读文件工具，允许模型读取工作区内指定文件内容。
- F3: myCode 应提供写文件工具，允许模型向工作区内指定路径写入完整文件内容。
- F4: myCode 应提供改文件工具，允许模型通过“原文唯一匹配替换”的方式修改文件。
- F5: 改文件工具在原文匹配不到或匹配多次时，应返回清晰、结构化的失败信息，而不是进行不确定修改。
- F6: myCode 应提供执行命令工具，允许模型在受控工作目录中执行命令并获取退出码、标准输出和标准错误。
- F7: myCode 应提供按模式找文件工具，允许模型通过文件名或路径模式发现相关文件。
- F8: myCode 应提供搜索代码内容工具，允许模型按文本或正则搜索工作区文件内容并获取匹配位置摘要。
- F9: myCode 应提供工具注册中心，集中登记所有可用工具，并支持按名称查找工具。
- F10: 工具注册中心应能把已登记工具转换成当前模型 Provider 可发送的工具声明列表。
- F11: LLM 客户端应支持模型流式响应中的工具调用事件，包括工具名称和 JSON 参数碎片。
- F12: LLM 客户端应能在流式过程中拼接工具调用参数碎片，并在工具调用结束时解析为完整 JSON 参数。
- F13: 当模型请求未知工具、参数不是合法 JSON、参数不符合工具约束或工具执行失败时，myCode 应把失败包装成结构化工具结果返回给模型。
- F14: 工具执行应有超时保护；超时时返回结构化失败结果，不能导致交互进程永久阻塞。
- F15: 工具执行结果应包含成功/失败状态、可读消息和必要的结构化数据，供模型判断下一步。
- F16: 工具执行完成后，myCode 应把工具结果追加到对话历史，并发起一次后续模型请求，让模型基于工具结果生成回复。
- F17: 本阶段一次用户输入最多触发一轮工具执行和一次工具结果回灌；模型拿到工具结果后的回复完成后，本轮流程停止。
- F18: 终端用户应能看到工具调用发生、执行成功或失败的可理解提示。
- F19: 文件类工具应限制在当前工作区或配置允许的根目录内，避免模型读写任意系统路径。
- F20: 命令执行工具应在当前工作区或配置允许的根目录内运行，并把命令失败作为工具失败结果返回，而不是直接中断 myCode。

## 非功能需求

- N1: 工具系统应与现有 Provider、会话和 CLI 分层保持清晰边界，避免把具体工具逻辑散落到 CLI 主循环中。
- N2: 工具接口和结果格式应便于后续扩展更多工具，而不需要重写现有工具调用流程。
- N3: 工具错误信息应清晰、可读、可供模型调整参数后重试。
- N4: 文件读写和命令执行能力默认应遵守工作区边界，避免越权访问。
- N5: 流式文本输出能力应继续可用；没有工具调用时，用户体验应与当前聊天流程一致。
- N6: 单次工具调用闭环应有自动化测试覆盖，包括成功路径和错误路径。
- N7: 示例和文档不得暗示本阶段已支持完整自主 Agent Loop。

## 不做的事

- 本阶段不实现多工具连续自动调用。
- 本阶段不实现任务级 Agent Loop、计划-执行-观察循环或自动直到完成。
- 本阶段不实现并发工具调用。
- 本阶段不实现用户交互式确认每次工具调用的权限系统。
- 本阶段不实现 Git 专用工具、网络搜索工具、浏览器工具或外部服务工具。
- 本阶段不实现补丁格式解析；改文件只支持原文唯一匹配替换。
- 本阶段不实现文件 diff 可视化、冲突合并或语义级代码编辑。
- 本阶段不实现跨会话持久化工具调用记录。
- 本阶段不绕过现有配置和 Provider 抽象去绑定单一模型供应商。

## 验收标准

- AC1: 工具系统中每个工具都能暴露名称、描述、参数约束和执行行为。
- AC2: 模型请求读文件工具时，myCode 能返回指定工作区文件内容；越界路径返回结构化失败。
- AC3: 模型请求写文件工具时，myCode 能写入工作区内目标文件；越界路径返回结构化失败。
- AC4: 模型请求改文件工具且原文在文件中唯一出现时，myCode 能完成替换并返回成功结果。
- AC5: 改文件工具遇到原文不存在或出现多次时，不修改文件，并返回明确失败原因。
- AC6: 模型请求执行命令工具时，myCode 能返回退出码、标准输出和标准错误；命令失败不会导致 myCode 崩溃。
- AC7: 命令执行超时时，myCode 返回结构化超时失败结果。
- AC8: 模型请求按模式找文件时，myCode 能返回匹配文件列表。
- AC9: 模型请求搜索代码内容时，myCode 能返回匹配文件、行号和内容摘要。
- AC10: 工具注册中心能登记六个核心工具，并能按名称查找。
- AC11: 工具注册中心能生成模型 Provider 可发送的工具声明列表。
- AC12: LLM 客户端能从流式响应中拼接工具调用 JSON 参数碎片，并解析为完整参数。
- AC13: 非法 JSON、未知工具、参数校验失败和工具执行异常都会被包装成结构化工具结果。
- AC14: 工具结果会被追加到对话历史，并触发一次后续模型请求。
- AC15: 一次用户输入最多执行一轮工具调用回灌；工具结果后的模型回复结束后不会继续自动调用下一轮工具。
- AC16: 不请求工具调用的普通聊天仍能保持现有流式文本输出和多轮上下文能力。
- AC17: 用户能在终端观察到工具调用名称、执行状态和失败原因。
- AC18: 自动化测试覆盖六个核心工具、注册中心、流式工具调用解析、工具结果回灌和单轮停止边界。
