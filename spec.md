# Mycode 五层权限系统 Spec

## 背景

Mycode 已具备五层权限系统，当前所有工具调用都会进入权限判定。在默认模式下，即使 Agent 只是使用专用工具读取文件、查找文件或搜索代码，也会因没有匹配规则而要求用户确认。这类只读操作不会修改工作区，却给日常项目观察带来不必要的交互成本。命令工具即使执行 `ls` 等查看命令，仍属于通用终端能力，不在本次自动放行范围内。

## 目标

- 让 `read_file`、`find_files`、`search_code` 三个专用只读工具无需权限规则、权限模式或人工审批即可直接执行。
- 写文件、编辑文件和执行终端命令仍使用统一的五层防御：不可绕过的危险命令黑名单、路径沙箱、可配置规则、权限模式和人在回路。
- 支持用户级、项目级、本地级与会话级规则，并按越靠近当前会话优先级越高的顺序判定。
- 支持严格、默认、放行三档权限模式，为规则未命中的调用提供一致行为。
- 交互审批提供“不同意、仅本次同意、本会话同意”三种选择，不提供永久同意。
- 审批界面使用上下方向键移动选项、回车确认，不要求用户输入字母命令。
- 权限拒绝以结构化工具结果回灌 Agent Loop，使模型可以调整方案，而不是直接终止任务。

## 功能需求

- F1: `read_file`、`find_files`、`search_code` 必须被视为专用只读工具，无需经过权限规则、权限模式或人工审批即可直接执行。
- F2: 专用只读工具仍必须遵守各自既有的参数校验和工作区路径边界；参数无效、绝对路径、`..` 越界或项目外符号链接不得因自动放行而获得访问能力。
- F3: `write_file`、`edit_file`、`run_command` 在实际执行前必须经过统一权限判定；未通过判定的调用不得触达工具实现。
- F4: `run_command` 始终按有副作用工具处理，即使命令内容是 `ls`、`pwd`、`cat`、`rg` 或 Git 查看命令，也不得使用专用只读工具的自动放行路径。
- F5: 系统必须在其他权限机制之前检查不可配置放开的危险命令黑名单，拦截灾难性文件系统破坏、磁盘覆写或格式化、系统目录递归权限修改、关机重启和 fork bomb。
- F6: 黑名单不得被权限模式、配置规则或人工审批覆盖；高风险但可能合理的开发操作不进入硬黑名单，交由后续权限层判定。
- F7: 专用文件工具产生的文件读取和写入必须限制在当前项目目录内；路径判定前必须解析规范路径和符号链接，防止通过绝对路径、`..` 或符号链接逃逸。
- F8: 命令工具必须检查命令文本中可识别的显式路径；显式路径必须经过规范化和符号链接解析，确认位于当前项目目录内后才可执行。命令运行时未在参数中显式表达的隐式文件访问不属于本阶段的强制隔离范围。
- F9: 系统必须支持 `工具名(模式)` 形式的 allow/deny 规则，模式支持精确匹配和 glob 匹配；专用只读工具即使存在历史 allow/deny 规则也直接执行，这些规则不参与只读调用判定。
- F10: 受控工具必须具有稳定的权限目标：命令工具匹配完整命令，写入和编辑工具匹配规范化后的工作区相对路径；内容和替换文本不参与匹配。
- F11: 规则分别来自用户级、项目级、本地级 YAML 和会话内存；优先级为“会话 > 本地 > 项目 > 用户”。
- F12: 跨层判定时，只采用最高优先级中存在匹配的规则层；同层多规则命中时，精确匹配优先于 glob，同等匹配类型下 deny 优先于 allow。
- F13: 系统必须支持 strict、default、allow 三档权限模式，作为受控工具在所有配置规则均未命中时的默认行为；权限模式不影响专用只读工具。
- F14: strict 模式拒绝未命中的受控工具调用；default 模式把未命中的受控工具调用交给用户确认；allow 模式放行未命中的受控工具调用。显式 deny 始终生效，黑名单与路径沙箱在所有模式下均不可绕过。
- F15: default 模式需要审批时，只提供“不同意、仅本次同意、本会话同意”三种选择，不得提供交互式永久同意。
- F16: 仅本次同意只适用于当前工具调用；本会话同意生成仅在当前进程有效的会话规则，重启后失效。
- F17: 人工审批通过可注入接口提供；交互式 CLI 必须展示工具名、权限目标和判定原因，并使用可视化选项列表收集用户决定。
- F18: 交互审批必须支持上下方向键移动当前选项并使用回车确认；不得要求用户输入 `d`、`o`、`s`、`p` 等字母后再提交。
- F19: 非交互环境没有可用审批接口时，待审批调用必须安全拒绝，不得默认放行。
- F20: 黑名单、沙箱或显式 deny 导致的拒绝不得再次请求用户覆盖。
- F21: 权限拒绝必须作为结构化工具结果写回对话历史，并继续 Agent Loop，让模型能够改用安全命令、专用工具或项目内路径。
- F22: 用户拒绝审批同样不得终止整个 Agent Loop；只有既有迭代上限、取消或其他既有停止条件可以结束任务。
- F23: 权限配置解析失败时必须给出可理解错误，且不得在配置含义不确定时降级为放行。
- F24: 用户仍可手工在项目本地权限 YAML 中编写 allow/deny 规则；移除交互式永久同意不得删除或禁用本地规则加载能力。

## 非功能需求

- N1: 受控工具的权限判定必须集中、确定且可测试，同一工具调用和同一规则集合应产生相同结果；专用只读工具的自动放行范围必须集中定义且可测试。
- N2: 默认行为必须遵循 fail closed；配置无效、显式路径解析失败、显式路径越界或审批不可用时不得自动执行。
- N3: 权限层应独立于具体 Provider，不因模型供应商或流式协议不同而改变安全语义。
- N4: 权限拒绝结果不得泄露配置文件中的敏感内容，也不得回显无需展示的大段文件内容或命令数据。
- N5: 用户审批提示必须清楚区分不可覆盖拒绝、规则拒绝和待用户决定，避免让用户误以为硬安全层可以放开。
- N6: 会话规则只在当前进程生命周期内有效；交互审批不得写入本地、项目或用户权限配置。
- N7: 多个有副作用的工具调用必须维持既有串行执行语义，避免并发审批或规则写入造成竞态。
- N8: 引入权限系统后，普通聊天、Plan Mode、Do Mode、工具结果回灌、取消和迭代上限等现有行为应保持可用。
- N9: 自动化测试必须覆盖黑名单、符号链接逃逸、规则优先级、匹配冲突、三档模式、仅本次与本会话两种交互放行范围、无永久同意、无交互安全拒绝以及拒绝后继续 Agent Loop。
- N10: 权限配置和错误信息应具备可读性，便于用户理解某次调用为何放行、拒绝或请求确认。
- N11: 专用只读工具自动执行不得削弱写入、编辑、终端命令、黑名单和工作区边界的既有保护。
- N12: 方向键菜单必须在用户取消、输入结束或终端交互不可用时安全拒绝，不得因界面异常自动放行。

## 不做的事

- 本阶段不识别或自动放行 `run_command` 中看起来只读的 shell 命令。
- 本阶段不新增其他自动放行工具；自动放行范围固定为 `read_file`、`find_files`、`search_code`。
- 本阶段不限制网络请求或按域名、协议、端口配置网络权限。
- 本阶段不实现 CPU、内存、进程数、执行时长之外的新资源配额。
- 本阶段不实现权限决策审计日志、历史查询或审计导出。
- 本阶段不提供图形化权限管理界面。
- 本阶段不自动把永久放行写入项目共享规则或用户全局规则。
- 本阶段不提供任何交互式永久同意入口，也不因审批结果写入 `.mycode/permissions.local.yaml`。
- 本阶段不删除手工维护用户级、项目级或本地级权限规则的能力。
- 本阶段不允许通过配置扩展、删除或覆盖灾难性命令黑名单。
- 本阶段不把普通高风险开发操作全部列入硬黑名单；它们由规则、模式和人工审批控制。
- 本阶段不新增网络沙箱或容器化运行环境。
- 本阶段不通过操作系统沙箱或容器拦截命令运行时的隐式文件访问；这一限制必须作为后续强化项保留在技术设计中。

## 验收标准

- AC1: 在 strict、default、allow 三种权限模式下，`read_file`、`find_files`、`search_code` 均无需审批即可执行。
- AC2: 专用只读工具存在匹配的 deny 规则、没有审批接口或处于 strict 模式时仍可直接执行。
- AC3: 专用只读工具使用无效参数、绝对路径、`..` 越界路径或项目外符号链接时仍被工作区边界拒绝。
- AC4: `run_command` 执行 `ls -la` 等查看命令时仍按受控工具处理；default 模式无匹配规则时必须请求审批。
- AC5: `write_file`、`edit_file`、`run_command` 仍在工具实现前产生明确权限判定，被拒绝时不进入工具实现。
- AC6: 代表性的根目录破坏、磁盘覆写或格式化、系统目录递归权限修改、关机重启及 fork bomb 命令均被硬拒绝。
- AC7: strict、default、allow 模式、所有层级的 allow 规则以及人工审批均不能放行黑名单命令。
- AC8: 高风险但可能合理的普通开发命令不因硬黑名单直接拒绝，而是继续进入规则、模式或审批判定。
- AC9: 写入和编辑工具使用绝对路径、`..` 越界路径或指向项目外部的符号链接时均被拒绝，且不会修改目标。
- AC10: 命令工具包含可识别的项目外显式路径、无法规范化的显式路径或指向项目外部的符号链接时不会执行；拒绝结果明确建议采用项目内路径或专用工具。
- AC11: 受控工具的精确规则只匹配完全一致的权限目标，glob 规则能匹配预期目标且不误匹配无关目标。
- AC12: `run_command(git *)` 按完整命令匹配；写入和编辑工具的规则按规范化项目相对路径匹配，写入内容和替换文本不改变规则结果。
- AC13: 会话、本地、项目、用户四层同时存在冲突规则时，受控工具的实际结果符合“会话 > 本地 > 项目 > 用户”。
- AC14: 同一层存在精确规则和 glob 规则时采用精确规则；同等匹配类型同时存在 allow 与 deny 时采用 deny。
- AC15: 受控工具的所有配置规则均未命中时，strict 模式拒绝、default 模式请求审批、allow 模式放行。
- AC16: 受控工具命中任意显式 deny 后都不会进入人工审批，也不会因 allow 模式而放行。
- AC17: default 模式的审批提示可观察到工具名、权限目标和请求原因，并只显示“不同意、仅本次同意、本会话同意”三个决定。
- AC18: 选择“本次放行”后只有当前受控工具调用获准，后续相同调用仍按正常流程重新判定。
- AC19: 选择“本会话放行”后，相同受控工具权限目标在当前进程内无需重复审批；重启进程后该规则消失。
- AC20: 审批菜单可以使用上、下方向键循环或移动选择，并使用回车确认；输入 `d`、`o`、`s`、`p` 字母不会作为审批决定提交。
- AC21: 完成交互审批后，`.mycode/permissions.local.yaml`、项目权限配置和用户权限配置均不因审批决定发生变化。
- AC22: 手工写入 `.mycode/permissions.local.yaml` 的合法 allow/deny 规则仍能在重新启动后加载并作用于受控工具。
- AC23: CLI 可以完成真实交互审批；取消、EOF、终端交互不可用或未提供审批接口时，会安全拒绝待审批的受控工具，但不阻止合法的专用只读工具。
- AC24: 黑名单拒绝、路径沙箱拒绝、显式规则拒绝和用户拒绝都以结构化工具结果回写对话历史。
- AC25: 权限拒绝后 Agent Loop 仍可进入下一次模型迭代，模型能够选择更安全的命令、专用工具或项目内路径继续任务。
- AC26: 多个有副作用工具调用继续串行判定和执行；多个专用只读工具可直接并发执行且不产生审批提示。
- AC27: 用户级、项目级和本地级 YAML 均可被加载；不存在的可选配置层不会导致启动失败；历史只读工具规则不会阻止专用只读工具执行。
- AC28: YAML 语法错误、未知权限结果、无效规则格式或未知工具名会产生可理解的配置错误，相关受控工具调用不会降级为放行。
- AC29: 普通聊天、Plan Mode 只读工具、Do Mode 完整工具集、用户取消和最大迭代次数等既有路径仍通过回归测试。
- AC30: 自动化测试覆盖专用只读工具自动执行、只读路径边界、不可绕过层、规则与模式判定、两种交互放行范围、方向键审批、无交互拒绝，以及拒绝后继续 Agent Loop 的端到端场景。
