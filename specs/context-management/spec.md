# MyCode 上下文管理 Spec

## 背景

MyCode 会在一次交互会话中持续保留用户消息、助手回复、工具调用和工具结果。随着 Agent 长时间工作，尤其是文件读取、搜索和命令输出等工具结果不断累积，请求上下文可能超过模型窗口，导致后续请求失败。当前系统虽然能够获得部分 API token usage，但尚未用它管理上下文预算。

## 目标

- 在每次模型 API 请求前自动控制单条工具结果和累计对话的上下文占用。
- 优先卸载体积较大的工具结果，尽量原文保留用户消息。
- 在对话逼近模型窗口上限时，通过结构化摘要压缩早期历史，同时保留近期原文。
- 让已卸载的工具结果可从磁盘重新读取，并明确阻止模型依据摘要臆测代码细节。
- 支持用户主动压缩，并在摘要连续失败时熔断，避免自动重试死循环。
- 本阶段使用 API usage 锚点结合字符增量的近似估算，不引入精确 tokenizer 或学习型摘要策略。

## 功能需求

- F1: MyCode 必须要求配置当前模型的上下文窗口上限。该值缺失、类型错误或不是正整数时，必须在启动阶段报告配置错误。

- F2: 轻量预防必须提供两个可选配置：单个工具结果阈值默认 8K token，同一工具调用批次的结果合计阈值默认 16K token；配置值必须为正整数。

- F3: 每次普通模型 API 请求前，MyCode 必须先执行轻量预防，再判断是否执行重量摘要。内部摘要请求不得递归触发新的压缩流程。

- F4: 单个工具结果超过阈值时，必须把截断前的完整原始结果写入当前会话的上下文文件；对话中只保留开头 1,000 字符、结尾 1,000 字符、文件路径、原始大小及卸载说明。

- F5: 同一工具调用批次的未卸载结果合计超过阈值时，必须按结果体积从大到小依次卸载，直到留在对话中的结果合计不再超过阈值。

- F6: 上下文文件必须存放在工作区内按会话隔离的隐藏目录，使模型可通过现有文件读取能力重新获取细节；MyCode 正常退出时必须删除本会话产生的上下文文件。

- F7: 上下文估算必须以最近一次成功 API 响应报告的输入 token usage 为锚点，只对锚点之后新增、删除或替换的内容使用字符数近似估算。尚无 usage 锚点时，必须对当前完整请求进行近似估算。

- F8: 自动模式下，预计请求量达到“上下文窗口上限减 13K token”时必须触发重量摘要。用户执行 `/compact` 时使用“窗口上限减 3K token”的目标预算，并立即执行轻量预防和重量摘要；该命令不得作为用户消息进入历史。

- F9: 重量摘要必须从历史尾部向前选择近期原文，保留约 10K token 且不少于 5 条消息。选择边界不得拆散一次助手工具调用及其对应工具结果。

- F10: 重量摘要必须把更早的可压缩历史整理为六个固定部分：用户目标与原始要求、已完成工作、关键决策与约束、相关文件及已卸载结果路径、当前问题与风险、待办与下一步。

- F11: 摘要请求必须明确禁止调用任何工具，并要求模型先生成分析草稿、再生成正式结构化摘要。只有正式摘要可以进入历史，分析草稿必须丢弃。

- F12: 摘要成功后，必须以结构化摘要替换较早的可压缩历史，保留选定的近期原文，并追加一条压缩边界消息，明确要求模型在需要文件或代码细节时重新读取对应文件，不得根据摘要臆测。

- F13: 用户原始消息应优先原文保留，不得由摘要模型改写。仅当保留它们会导致请求无法满足预算时，才允许将最早的用户消息原文写入会话文件，并在历史中保留路径和边界说明。

- F14: 摘要请求发生 API 错误、输出缺少规定部分、输出不可解析或压缩后仍未达到目标预算时，必须视为失败；失败不得留下半更新的历史。连续失败次数在成功后清零。

- F15: 自动摘要连续失败 3 次后，本会话必须停止自动摘要，避免重复调用。用户仍可通过 `/compact` 主动重试；手动摘要成功后清零失败次数并恢复自动摘要。

- F16: `/compact` 必须向用户显示压缩前估算量、压缩后估算量、被卸载或摘要的范围以及成功、失败或无需压缩状态。

- F17: 摘要失败、熔断或压缩不足后，如果上下文仍超过本次允许预算，MyCode 不得发送普通模型请求；必须报告当前估算量、预算和原因，并提示用户执行 `/compact` 重试。

## 非功能需求

- N1: 上下文管理必须兼容现有 OpenAI、Anthropic 和 DeepSeek Provider，不得改变正常消息、工具调用和工具结果在各协议中的合法顺序。

- N2: Token 估算必须覆盖实际发送请求中的系统提示、动态提示、工具定义和对话消息；算法应确定、可测试，并在无法准确判断时偏向保守估算。

- N3: 轻量卸载和重量摘要对历史的更新必须具备原子性。文件写入、摘要调用或结果校验失败时，不得留下只更新了一部分的对话历史。

- N4: 上下文文件只能写入当前工作区指定的会话目录，不得越过工作区边界或跟随符号链接写到外部；文件名必须避免工具调用之间相互覆盖。

- N5: 上下文文件内容不得直接打印到终端。错误和状态信息可以显示路径、大小和失败阶段，但不得额外暴露原始工具结果或已卸载用户消息。

- N6: 摘要请求在所有 Provider 上都必须以空工具集合发送；即使模型输出类似工具调用的内容，也不得执行。

- N7: 压缩不得产生孤立的工具结果、缺失对应调用的工具消息或其他 Provider 无法接受的消息序列。

- N8: 文件写入失败、文件清理失败、摘要 API 失败、摘要格式错误和压缩不足必须具有可区分的用户可见信息；正常退出时清理失败应告警，但不得阻止进程退出。

- N9: 自动压缩不得形成递归调用或无界重试；一次请求准备过程最多发起一次自动摘要，连续失败熔断状态必须限定在当前会话。

- N10: 新能力不得改变现有六个内置工具、MCP 工具、权限系统、Plan/Do 模式和流式输出的既有行为，除非工具结果因本规格要求被卸载。

- N11: 自动化测试必须覆盖三类 Provider 的消息合法性、两级轻量卸载、估算锚点、自动与手动摘要、近期消息边界、用户原文保护、摘要失败熔断、超预算拒绝和会话文件清理。

## 不做的事

- 本阶段不引入精确 tokenizer，也不按不同模型维护 tokenizer 词表。
- 本阶段不使用机器学习、历史反馈或在线实验自动优化摘要策略、阈值或保留范围。
- 本阶段不自动查询 Provider 或外部服务来推断模型上下文窗口；窗口上限必须由用户配置。
- 本阶段不实现跨进程的对话恢复、摘要恢复或会话续接。
- 本阶段不把上下文文件作为永久日志、审计记录或项目知识库。
- 本阶段不实现向量检索、语义检索、自动召回或基于相关性的消息选择。
- 本阶段不增加图形界面、上下文浏览器或文件管理界面。
- 本阶段不允许为摘要单独选择另一家 Provider 或另一套凭据；摘要沿用当前会话的模型服务。
- 本阶段不自动总结或改写系统提示、工具定义和当前保留区内的近期消息。
- 本阶段不保证异常崩溃、进程被强制终止或机器断电时完成会话文件清理。
- 本阶段不解决模型服务本身返回错误 usage、错误上下文窗口声明或不遵守摘要格式指令的问题；系统只负责校验、失败处理和熔断。

## 验收标准

- AC1: 合法的上下文窗口配置和默认轻量阈值可成功加载；窗口上限缺失或非法、任一可选阈值非法时，启动失败并指出对应配置项。（覆盖 F1、F2）

- AC2: 一个超过 8K token 默认阈值的工具结果产生后，磁盘文件包含截断前的完整原始结果；下一次请求中的对应工具消息只包含首尾各 1,000 字符、可读取路径、原始大小和卸载说明。（覆盖 F3、F4）

- AC3: 同一批次存在多个均未超过单项阈值、但合计超过 16K token 的工具结果时，系统按体积从大到小卸载；达到合计阈值后停止，较小且仍可容纳的结果继续保留原文。（覆盖 F5）

- AC4: 两个并发或连续工具结果不会覆盖彼此的上下文文件；模型能在当前会话中通过保留路径重新读取完整内容；正常退出后本会话目录被删除。（覆盖 F6、N4）

- AC5: 首次请求没有 usage 时按完整请求估算；获得成功响应的 input usage 后以该值为锚点；随后新增、删除或替换内容时只估算相对锚点的变化，并把系统提示和工具定义的变化计入结果。（覆盖 F7、N2）

- AC6: 普通请求预计达到窗口上限减 13K token 时，系统先完成轻量预防，再最多发起一次自动摘要；未达到阈值时不调用摘要模型。（覆盖 F3、F8、N9）

- AC7: 输入 `/compact` 时，该文本不会出现在模型消息历史中；即使尚未达到自动阈值，只要存在可压缩的早期历史，也会按窗口上限减 3K token 的目标执行压缩，并展示前后估算量、处理范围和结果；没有可压缩历史时明确显示无需压缩。（覆盖 F8、F16）

- AC8: 重量摘要后，尾部至少保留 5 条近期原文，并在历史足够时保留约 10K token；任何助手工具调用及其对应结果都不会被边界拆开。（覆盖 F9、N7）

- AC9: 摘要请求携带空工具集合，提示中明确禁止工具调用，并要求先产出分析草稿再产出正式摘要；最终历史只包含具有六个规定部分的正式摘要，不包含分析草稿。（覆盖 F10、F11、N6）

- AC10: 摘要成功后，早期可压缩历史被结构化摘要替换，近期消息保持原文，并紧接一条边界消息，要求重新读取文件细节且不得根据摘要臆测代码。（覆盖 F12）

- AC11: 正常压缩时，位于早期范围内的用户消息仍按原文保留；只有它们导致目标预算无法满足时，最早的用户消息才会逐条原文存盘，历史只留下路径和说明，摘要中不存在模型改写后的替代文本。（覆盖 F13）

- AC12: 模拟文件写入失败、摘要 API 失败、摘要缺少规定部分、摘要不可解析和摘要后仍超目标预算时，压缩均报告失败，原有历史保持完整且可继续检查，不出现部分替换状态。（覆盖 F14、N3、N8）

- AC13: 自动摘要连续失败三次后，后续普通请求不再自动调用摘要模型；执行 `/compact` 仍会主动重试，成功后恢复自动摘要并从零重新计算连续失败次数。（覆盖 F15）

- AC14: 摘要失败、熔断或压缩不足后，如果估算量仍超过本次预算，普通模型 API 不会收到请求；终端显示当前估算量、目标预算、失败原因和 `/compact` 重试提示。（覆盖 F17）

- AC15: OpenAI、Anthropic 和 DeepSeek 请求在压缩前后都保持各自协议要求的角色顺序和工具调用关联；摘要请求在三类 Provider 上均不携带工具定义。（覆盖 N1、N6、N7）

- AC16: 正常退出时清理失败会产生不含原始内容的警告，但退出流程仍完成；终端上的压缩状态和错误不会打印已卸载工具结果或用户消息正文。（覆盖 N5、N8）

- AC17: 现有内置工具、MCP 工具、权限规则、Plan/Do 模式、流式文本和 token usage 展示通过回归测试。（覆盖 N10）

- AC18: 端到端长会话中连续产生一个超大工具结果、一个合计超限的多结果批次以及足以触发重量摘要的历史后，普通任务仍能继续：完整结果可按路径重读，早期历史形成六段摘要，近期消息保持原文，请求估算低于自动预算，退出后会话文件被清理。（覆盖 F1-F17、N1-N11）
